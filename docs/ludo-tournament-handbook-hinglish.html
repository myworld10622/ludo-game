Project Overview

Ye project 3 major parts me split hai, jo milke ek complete tournament platform banate hain:

🖥 Laravel Backend
REST API, Admin Panel, User Panel, Tournament Engine, Reports, Support/Tickets system
⚡ Node.js
Socket / Realtime layer — Ludo room events aur live gameplay connectivity handle karta hai
📱 Unity
Player-facing mobile game app — Login, Wallet, Tournament browse, Register, aur Play flow

🔄 Simple Flow — End to End

1
User Unity app me login karta hai
2
Unity Laravel API se tournament list aur user data fetch karti hai
3
Tournament join/register Laravel me hota hai — wallet deduction bhi isi step me
4
Jab match/table assign hoti hai to Unity Socket/Node flow se room join karti hai
5
Match ka result backend me settle hota hai — prize distribute hota hai
6
Admin aur User dono panels me reports show hoti hain

03 ♻️

Tournament Lifecycle & Statuses

Laravel me tournament ka ek defined status flow hai. Har status ka ek specific matlab hai:

Status Matlab (Meaning) Kab aata hai
draft Tournament create hua hai, lekin live nahi — registration ke liye open nahi Tournament create hone par by default
registration_open User register kar sakta hai — entry fee accept ho rahi hai Approved ho aur registration start time aa jaye
registration_closed New registration band — existing players next stage ke liye wait ya claim kar sakte hain Registration end time cross ho jaye
in_progress Matches chal rahe hain — tournament live hai Match/play phase shuru ho
completed Tournament finish — winners/payout/report final ho gaye Last match complete ho
cancelled Tournament cancel — eligible phase me refund ho sakta hai Admin manually cancel kare

⚙️ Scheduler Automation

  • draft → registration_open   jab tournament approved ho aur registration start time aa jaye
  • registration_open → registration_closed   jab registration end time cross ho jaye
  • No-show users ko slot miss hone par disqualify kiya ja sakta hai

04 🆕

Tournament Creation Rules

🛡 Admin Tournament — Full Control

Admin ke paas sabhi advanced options available hain:

Name Type Format Bracket Mode Entry Fee Max Players Players/Match Platform Fee % Dates Description Terms Prize Split Play Slots Bot Options
👤 User Tournament — Restricted Flow

Intentionally restricted — fraud prevention ke liye:

  • Tournament create kar sakta hai ✅
  • Self-approve nahi kar sakta ❌
  • Fake registrations add nahi kar sakta ❌
  • Force live nahi kar sakta ❌
  • Bot options direct nahi use kar sakta ❌
  • Real tournament setup karta hai mostly ✅
⚠️ User-created tournament admin review queue me jata hai — admin detail review, edit, approve ya reject with reason kar sakta hai

🔍 Admin Review Actions (User Tournament ke liye)

  • Tournament detail review — sab fields dekhna
  • Tournament edit karna — incorrect fee/timing/prize fix karna
  • Approve karna — tournament live queue me daalna
  • Reject with reason — user ko message ke saath reject karna
  • Support thread me user ko directly message karna

06 🎯

Tournament Join / Register Flow

📋 6.1 Unity Tournament List — Kaunse Tournaments Dikhte Hain

Unity tournament panel me current approved public tournaments show hote hain:

registration_open registration_closed in_progress

📲 6.2 Registration Open Tournament — User kya Karta Hai

Important: Direct table par nahi jata — pehle details popup open hota hai

Popup me show hota hai:

• Tournament status
• Current players count
• Entry fee amount
• Prize pool
• Registration close time
• Tournament start time
• Play slots schedule
• User's wallet balance

Agar user Register click karta hai — ye steps hote hain:

1
Backend registration API hit hoti hai
2
Wallet balance check hota hai — insufficient balance par fail
3
Entry fee deduct hoti hai — same transaction me
4
Registration record create hota hai — user registered state me
5
Unity ka local wallet balance update hoti hai

⏳ 6.3 Already Registered — Tournament Not Live Yet

  • User ko detail popup me clearly indicate hota hai: "You are already registered"
  • Message: "Wait for tournament to go live"
  • User double-register nahi kar sakta — idempotent registration logic hai

▶️ 6.4 Tournament In Progress

  • Agar user registered hai aur tournament in_progress hai
  • Unity "Play Now" button allow karti hai
  • Claim room / match flow trigger hota hai

🔒 6.5 Full Tournament

  • Jab max players limit reach ho jaye — new registration blocked
  • User tournament details dekh sakta hai
  • Register button disabled / unavailable ho jata hai

07 💰

Wallet Rules

  • Registration time par wallet se entry fee instantly deduct hoti hai
  • Duplicate registration idempotent hai — same registration return hoti hai, double deduction nahi
  • Insufficient balance par registration fail hota hai — koi partial deduction nahi
  • Cancel registration ke case me eligible phase me refund hota hai
  • Tournament complete hone par winners ke wallet me prize amount credit hota hai
  • Platform fee transaction bhi alag se record hoti hai
⚠️
Critical Point: Tournament register API wallet deduction same transaction me karti hai — isliye successful register ka matlab entry fee already deduct ho chuki hai. Agar user register confirm ho gaya to fee gai.

Bot Rules

🎮 8.1 Public Normal Ludo Queue (Non-Tournament)

  • Bots allowed ho sakte hain — configuration ke hisaab se
  • Bot fill after timeout ka logic available hai
  • Room me minimum real players meet ho jaye to remaining seats bots se fill ho sakti hain

Bot Decision Logic:

✓
Room full real players se ho gaya → Start immediately
?
Bots allowed hain AND minimum real player count meet hua → Bots fill karke start
⏳
Conditions meet nahi hui → Wait karo

🏆 8.2 Tournament Bot Rules — Do Alag Concepts

📋 Concept 1: Registration-Level Bots
  • Admin tournament me bots ko registered participants ke roop me add kar sakta hai
  • Max bot percentage rule hai
  • User panel se bots add nahi hote
🔴 Concept 2: Live Room Fill Bots
  • Current tournament room provisioning: allow_bots = false
  • Live room auto bot fill active nahi hai
  • Provisioned tournament rooms real-human seats hi use kar rahe hain
🚨
Critical Gap — Bot Intent vs Room Reality:
Tournament model me bot_allowed = true ho sakta hai — but current provisioned tournament Ludo room code real-human seats hi use kar raha hai.

Matlab: Room creation time par tournament room me live bot auto-fill active nahi hai.
Agar business requirement hai ki missing players ki jagah bots fill ho to — yeh currently implement nahi hai.

Real User Join Conditions

📝 Tournament Register ke liye — Sabhi Conditions
  • Tournament approved ho
  • Status: registration_open ho
  • Tournament full na ho (max players reached nahi)
  • Wallet me sufficient balance ho (entry fee cover karne ke liye)
  • User already refunded registration state me na ho
▶️ Room/Table Claim ke liye — Sabhi Conditions
  • User registered ho us tournament me
  • Active play slot chal rahi ho (agar slots defined hain)
  • Claim request valid tournament/entry ke liye ho
⚠️ Active slot nahi chal rahi → Claim reject:
"Tournament can only be played during scheduled play slots."
10 🕐

Play Slot Rules

🎯 Play Slots ka Purpose
  • Users ko pehle se bata diya jaye ki kis time online rehna hai
  • Table fill aur play coordination better ho
  • Random late play avoid ho
  • Tournament schedule disciplined rahe

⚙️ Configuration

  • Tournament create/edit form me 1 se 5 slots define ho sakte hain
  • Slot details report pages par visible hain
  • Unity tournament popup me visible hain
  • Claim room ke waqt active slot validation hoti hai

❌ 10.1 Agar User Slot Me Play Nahi Karta

1
Scheduler every minute run ho sakta hai
2
Slot end hone ke baad — registered/checked_in players inspect hote hain
3
Jin logon ne us slot me check-in/claim nahi kiya — unka registration status:
4
Status update: registered ya checked_in → disqualified

📍 10.2 Check-in Tracking — Kya Save Hota Hai

Claim room successful hone par registration record me ye fields save hote hain:

  • last_checked_in_at — exact timestamp jab user ne room claim kiya
  • last_checked_in_slot_index — kaunse slot number me aaya
ℹ️ Iska fayda: Admin dekh sakta hai ki user slot me aaya ya nahi aur kaunse slot me aaya — disqualification decisions is data par based hote hain

11 ⚔️

Tournament Match / Table Behavior

🔧 11.1 Current Provisioning Logic — Kaise Kaam Karta Hai

  • Tournament entries/registrations se bracket generate hota hai
  • Bracket ke basis par match entries assign hote hain
  • Room provisioning service round-wise rooms create karti hai
  • Provisioned room me all seeded human entries mapped hote hain
  • Tournament room me: allow_bots = false
🚨
11.2 Important Consequence — Attendance vs Provisioning Gap:

Current backend ke hisaab se room creation ka decision sirf "is match me kaunse seeded entries hain" par based hai — "abhi exactly kitne users online hain" par nahi.

Matlab: Agar 4-player match ke 4 seeded participants hain → room provision ho sakta hai — chahe us moment sirf 1 ya 2 users actually online hon.

Missing players ke liye current room provisioning live bot fill nahi karti. Practical match success online attendance par depend karega.

Scenario Explanations

Ye sabhi real-world scenarios hain jo tournament flow me encounter ho sakte hain:

Scenario 12.1
🔍 Registration Open — User Details Dekhta Hai
User tournament card click karta hai → details popup open hota hai → fee, prize, slots, timing check karta hai → wallet sufficient hai to register kar sakta hai
Scenario 12.2
✅ User Register Karta Hai
Wallet deduction instantly → registration create → user registered state me chala jata hai → Unity wallet update hoti hai
Scenario 12.3
⏳ Registered Hai, Tournament Start Nahi Hua
User wait karega → details me registered state dekh sakta hai → tournament live hone par "Play Now" option aayega
Scenario 12.4
❌ Play Slot Me User Online Nahi Aaya
User room claim nahi karega → slot end hone ke baad scheduler inspect karega → disqualify kar sakta hai → tournament se effectively out
Scenario 12.5
🤖 Bots Allowed Nahi Hain
Tournament room me current code bots disable kar raha hai → missing real players ki jagah live room auto-bot fill nahi hoti → user attendance bahut important ho jati hai
Scenario 12.6
⚠️ 4-Player Tournament, Sirf 1 User Online
Bracket/provisioning attendance se pehle ho sakti hai → room pre-created → but actual gameplay sabke online hone par dependent → absent users disqualification risk me
Scenario 12.7
⚠️ 4-Player Tournament, Sirf 2 Users Online
Same as 12.6 — tournament-specific live bot fill active nahi → absent players ka no-show outcome scheduler / later match handling par depend karega
Scenario 12.8
🔒 Tournament Full Ho Gaya
New user register nahi kar sakta → existing participants tournament wait / live stage ke liye continue karte hain
Scenario 12.9
🏆 Tournament Completed
Final match complete → winners ranked → prize payout real users ke wallet me credit → reports user/admin panel me visible ho jati hain

19 💡

Practical Recommendations

✅ Real-Player-First Approach (Current — Safer)

Agar aap real-player-first tournament chalana chahte ho:

  • User ko slots clearly dikhaye jaye
  • Slot miss karne par disqualification rahe
  • Tournament room me bots default off rahein
  • Admin special cases manually monitor kare
✅ Current system is approach ke liye already ready hai
🤖 Auto-Bot-Fill Approach (Needs New Build)

Agar chahte ho ki 4-player table me 1-2 real users aayein aur baaki bots se start ho:

  • Ye abhi fully active behavior nahi hai
  • Dedicated tournament-bot-start policy implement karni hogi
  • Room provisioning service me additional logic chahiye
⚠️ Ye feature alag explicitly build karna padega — current code me nahi hai
