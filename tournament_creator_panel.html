Core Concept

Tournament creation is moved entirely off the Unity app and onto a dedicated web panel. Unity's role is only to detect intent → verify email → show the user their web panel URL. Everything else (creating, configuring, monitoring) happens in the browser.

Unity does

  • Check email verified
  • Show web panel URL popup
  • Send info email

Web panel does

  • Tournament creation form
  • List user's tournaments
  • Players / Matches / Bracket

No change needed in

  • Node.js socket layer
  • Unity tournament list UI
  • Existing API routes

Why this approach is better

Web forms are far better for creating tournaments — full keyboard, larger screen, easier to fill prize structure and settings. No new Unity UI to build or maintain. The web panel also works from any browser on any device forever, not just in-app. Simpler backend: reuse existing user auth, no per-tournament tokens or temp passwords to manage.

Email trigger timing

The email is sent when the user first taps "＋ Create" and their email is verified — before they create any tournament. It's a "welcome to your panel" email, not a "tournament created" email. This way the user gets the URL immediately and can bookmark it. The email is only sent ONCE per user (check if already sent via a users.panel_email_sent_at timestamp — if null, send; if already sent, skip).

Security: user_id in URL vs. random token

Using the numeric user_id in the URL is fine because the password is the actual security. The URL identifies which panel to show; the login gate protects the data. Anyone who navigates to the URL without the password just sees a login form. If we want extra obscurity, we can use uuid instead of numeric ID — decision in Open Questions.

Auth uses existing users table

Login checks: users table where email = input AND Hash::check(input_password, users.password). The user's existing app password works — no new password column, no temp passwords. If the user registered via social login (no password set), they'll need to set one from their profile in the app first (note this in the login page footer).

Bracket Tab

Rounds displayed as columns of match cards. Each card shows two players and the winner (if decided). V1: tabular layout. V2: visual bracket with connecting lines (optional enhancement).

Read-only panel — no admin override

Creator panel is monitoring only. Creators can see everything but cannot force-set winners, kick players, or change tournament settings after creation. Those powers remain exclusively in the admin panel at /admin.

Authentication

// CreatorPanelController::login()
$user = User::where('email', $request->email)->first();
if (!$user || !Hash::check($request->password, $user->password)) {
    return back()->withErrors(['email' => 'Invalid credentials.']);
}
// Verify this user owns this panel URL
if ($user->id !== (int) $request->route('userId')) {
    return back()->withErrors(['email' => 'Invalid credentials.']);
}
session(['creator_panel_user_id' => $user->id]);
return redirect("/panel/{$user->id}/tournaments");

// CreatorPanelAuth middleware
public function handle($request, $next)
{
    $sessionUserId = session('creator_panel_user_id');
    $routeUserId   = (int) $request->route('userId');
    if (!$sessionUserId || $sessionUserId !== $routeUserId) {
        return redirect("/panel/{$routeUserId}/login");
    }
    return $next($request);
}

New Unity API endpoint (notify-panel)

// POST /api/v1/user/notify-panel
// Called by Unity when user taps Create with verified email
// Auth: Bearer token (existing user auth)
public function notifyPanel(Request $request): JsonResponse
{
    $user = $request->user();
    if (is_null($user->panel_email_sent_at)) {
        Mail::to($user->email)->queue(new CreatorPanelWelcomeMail($user));
        $user->update(['panel_email_sent_at' => now()]);
    }
    return response()->json([
        'panel_url' => url('/panel/' . $user->id),
    ]);
}
