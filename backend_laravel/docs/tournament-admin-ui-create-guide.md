# Tournament Admin UI Create Guide

This guide explains exactly what to fill when creating a tournament from the admin UI so that:

- it appears in the Unity tournament popup
- it is actually joinable
- it transitions correctly into gameplay

## Core Rule

A tournament is visible and joinable in Unity only when all of these are true:

- `status` is `published` or `entry_open`
- `entry_open_at` is `null` or is in the past
- `entry_close_at` is `null` or is in the future

Important:

- `published` does not automatically mean "joinable now"
- if `entry_open_at` is in the future, Unity should not treat it as joinable
- `live` is an old/legacy status and should not be used for the new tournament flow

## Recommended Status Flow

Use this status lifecycle:

1. `draft`
2. `published`
3. `entry_open`
4. `entry_locked`
5. `seeding`
6. `running`
7. `completed`

Notes:

- `published` means the tournament exists and can become open
- `entry_open` means players can join now
- `entry_locked` means no more entries
- `seeding` and `running` are match-execution stages

## Safe Defaults For New Tournament

For a standard new Ludo tournament, use:

- `game_id`: Ludo game ID
- `status`: `published`
- `tournament_type`: `standard`
- `entry_fee`: as needed, `0` allowed
- `max_entries_per_user`: `1` for single-ticket tournaments, `>1` for multi-ticket tournaments
- `max_total_entries`: bracket size target, for example `64`, `128`, `256`
- `min_players`: `2`
- `match_size`: `2` or `4`
- `advance_count`: `1`
- `currency`: `chips` or your configured value
- `registration_starts_at`: `now` or a past time
- `registration_ends_at`: a future time before tournament start
- `starts_at`: after registration end
- `prize_slabs`: at least one valid prize row

## What Each Important Field Means

### Identity fields

- `code`
  - unique tournament code
  - auto-generated by backend if left blank in admin flow

- `name`
  - user-facing tournament name
  - example: `Gold Ludo 256`

- `slug`
  - unique URL/admin identifier
  - auto-generated by backend if left blank in admin flow

### Status fields

- `status`
  - use `draft` while editing
  - use `published` when ready
  - do not use `live` for new tournament setup

### Entry settings

- `entry_fee`
  - fee per ticket

- `max_entries_per_user`
  - how many tickets one user can buy
  - example:
    - `1` = single entry only
    - `10` = one user may buy 10 tickets

- `max_total_entries`
  - total tickets allowed in this tournament
  - this is entry-based, not user-based

- `min_players`
  - minimum entries needed to run
  - recommended: `2`

### Match format

- `match_size`
  - `2` or `4`
  - this decides whether each table is 2-player or 4-player

- `advance_count`
  - recommended current value: `1`
  - one winner advances from each match

### Time fields

- `registration_starts_at`
  - maps to `entry_open_at`
  - players cannot join before this time

- `registration_ends_at`
  - maps to `entry_close_at`
  - players cannot join after this time

- `starts_at`
  - tournament seeding/start execution time
  - should usually be after registration end

- `ends_at`
  - optional expected end time

## Most Common Admin Mistakes

### Mistake 1: status is `published` but join still fails

Cause:

- `registration_starts_at` / `entry_open_at` is in the future

Result:

- backend error: `Tournament entries are not open yet.`

Fix:

- set registration start to current time or earlier

### Mistake 2: tournament not visible in Unity

Cause:

- status is not joinable
- or registration start is in the future
- or registration end is already past

Fix:

- use:
  - `status = published`
  - `registration_starts_at <= now`
  - `registration_ends_at >= now`

### Mistake 3: admin sets status to `live`

Cause:

- old legacy assumption

Result:

- new public tournament list may not treat it correctly for join flow

Fix:

- use `published` / `entry_open` instead

### Mistake 4: no prize slabs

Cause:

- create validation requires at least one prize row

Fix:

- add at least one slab, for example:
  - rank 1 to 1
  - fixed
  - amount 100

## Fields Removed From Admin Form

These fields were removed from the manual admin form because they were duplicate or engine-derived:

- `bye_count`
- `joined_players`
- `max_players` snapshot
- `rule players_per_match`
- `rule advance_count`
- manual `ticket_prefix`

The admin should now mainly fill:

- tournament identity
- pricing
- entry limits
- match format
- bracket size
- registration window
- start time
- prize slabs

## How Unity Decides Joinability

Unity now follows backend `can_join`.

That means:

- if backend says `can_join = true`
  - button shows `Join Tournament` or `Play Tournament`
- if backend says `can_join = false`
  - button shows `Entries Closed`

## Operational Recommendation

For admin UI, prefer this create workflow:

1. Create tournament as `draft`
2. Fill all times carefully
3. Save
4. Publish only when registration window is correct
5. Verify via API or Unity popup

## Quick Checklist

Before expecting a tournament to show and join in Unity:

- `game_id` is Ludo
- `status` is `published`
- `registration_starts_at <= now`
- `registration_ends_at > now`
- `starts_at >= registration_ends_at`
- `match_size` is `2` or `4`
- `advance_count` is `1`
- at least one prize slab exists
- `max_total_entries` is filled
- `max_entries_per_user` is correct

## Current System Limitation

If admin creates tournaments using old legacy status conventions like `live`, behavior may be inconsistent with the new tournament flow.

For the current production path, use:

- `draft`
- `published`
- `entry_open`
- `entry_locked`
- `seeding`
- `running`
- `completed`
