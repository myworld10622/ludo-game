1. Project Overview & Current Status

The Ludo Tournament System is built on top of an existing real-money Ludo game. It adds structured multi-player tournament support with brackets, prize pools, bot fallback, private invite-only rooms, and user-created tournaments.

1.1 What Is Currently Live (Implemented)

✅ Tournament CRUD (create, publish, close, cancel, generate bracket)
✅ Player registration (idempotent, wallet deduction ready)
✅ Knockout + Round Robin bracket generation
✅ Queue-style matchmaking when no bracket (first player creates room, next fills it)
✅ Node.js socket room claim flow (ludo.tournament.claim_room)
✅ Bot fill via scheduleBotFill — one bot per 8 seconds per empty seat
✅ Match result recording + bracket advancement
✅ Prize distribution to wallets on tournament completion
✅ Private tournament with invite code + optional password
✅ User-created tournaments (public & private) via Unity Create Tournament panel
✅ Unity tournament list panel — compact fee-table style rows
✅ Unity Create Tournament panel — instant / scheduled modes
✅ Simulation command: php artisan tournament:simulate

3. Tournament Lifecycle — Complete Flow

3.1 Admin-Created Tournament (Bracket Mode)

1
Create: Admin creates tournament via POST /api/v1/tournaments with name, format, entry fee, max_players, prize structure (must sum 100%). Status = draft.
2
Publish: Admin calls POST /{tournament}/publish → status = registration_open. Tournament appears in Unity lobby.
3
Players Register: Unity calls POST /{tournament}/register (idempotent — safe to retry). Entry fee checked and deducted from wallet. Registration stored with status=registered.
4
Close Registration: Admin calls POST /{tournament}/close-registration → status = registration_closed. Prize pool auto-calculated (entry_fee × players × 80%).
5
Generate Bracket: Admin calls POST /{tournament}/generate-bracket → TournamentBracketService creates all TournamentMatch rows with assigned players. Status = in_progress.
6
Player Claims Room: Unity calls POST /{tournament}/claim-room → Laravel finds the player's current unplayed match → returns room data with human players only.
7
Socket Connect: Unity emits ludo.tournament.claim_room on /ludo_v2 namespace → Node.js builds room state from claim-room data. Bot fill timer starts for empty seats.
8
Game Starts: When room full (or all bots filled), Node.js calls POST /api/internal/v1/ludo/rooms/{id}/start → Laravel detects tournament-match-{id} prefix, updates TournamentMatch status = in_progress.
9
Match Played: Ludo game runs via existing game engine. Seats/tokens controlled by Node.js server-side logic.
10
Match Result: Node.js calls POST /api/internal/v1/tournaments/matches/{match}/result with scores, positions, winner. Laravel records result, marks loser as eliminated, advances winner to next match in bracket.
11
Next Round: Winner is re-inserted into next-round TournamentMatch. When that match fills, it becomes waiting. Winner calls claim-room again → new socket room.
12
Final / Completion: When all matches complete, tournament status = completed. Prize distribution runs automatically: wallets credited, TournamentWalletTransaction rows created.

3.2 User-Created Tournament (Queue Mode)

User-created tournaments work the same as admin ones, except:
• User creates via Unity Create Tournament panel (instant or scheduled)
• Status starts as registration_open immediately (instant mode) or at schedule time
• If no bracket is generated, queue-mode matchmaking is used: first player creates a waiting match, next player fills it, etc.
• Bracket can be generated by admin later, or left in queue mode for smaller private groups.

3.3 Queue-Mode Matchmaking (No Bracket)

When a player claims a room and there is no pre-generated bracket, findOrCreateWaitingMatch() is called:

  • Looks for an existing waiting/scheduled match with empty slots.
  • If none found, creates a new TournamentMatch with status waiting.
  • Adds the player as a TournamentMatchPlayer.
  • Node.js starts bot fill timer — bots fill empty seats after 8s each.
  • When all seats filled → game starts automatically.

5. Bot Behavior & Fill Rules

5.1 How Bot Fill Works

Rule: Bots are NEVER pre-seated in the room. The claim-room API returns only human players. Node.js receives empty seats and schedules bot fill after bot_fill_after_seconds (default: 8 seconds).
  • When a tournament room is created in Node.js (buildTournamentRoomFromClaim), minRealPlayers = 1 is set.
  • If allowBots = true AND currentPlayers < maxPlayers → scheduleBotFill(room) is called.
  • After 8 seconds: engine.buildStartDecision(room) checks if realPlayers ≥ minRealPlayers (1). If yes → adds one bot seat.
  • State changes to waiting_bot_fill and emits a snapshot. Unity client shows "Waiting for X more players..."
  • Timer reschedules itself for the next bot until room is full → game starts.
  • Max bot % rule: max_bot_pct = 5% stored in tournament. Enforcement is at admin/registration level; bot fill itself doesn't recheck this at runtime.

5.2 Bot Rules for Prizes

  • Bots are registered as is_bot = true in TournamentRegistration.
  • During distributePrizes(), the loop skips bot registrations: prize cascades to the next real player.
  • Bots are excluded from financials (is_bot = false filter on entry fee stats).

5.3 Bot Flow Sequence

User joins → claim-room → Node.js gets {players: [human_only]}
 → scheduleBotFill() → state = "waiting_bot_fill"
 → after 8s: buildStartDecision() → shouldFillBots = true
 → push bot seat → emit ludo.room.bot_joined to all clients
 → Unity shows bot joining animation (via OnBotJoined in LudoV2MatchmakingBridge)
 → if room full → startRoom() → ludo.room.starting emitted
 → Unity enters game board

8. Node.js Socket Events (Real-time)

Namespace: /ludo_v2 at BaseSocketUrl/socket.io/

8.1 Normal Match Events

EventDirectionDescription
ludo.queue.joinClient → ServerJoin a fee-based room queue. Payload: {userId, roomUuid, maxPlayers, entryFee, allowBots, ...}
ludo.room.waitingServer → ClientRoom snapshot while waiting for players. Contains seats[] array.
ludo.room.bot_joinedServer → AllA bot was added to a seat. Payload: {room_id, seat: {seatNo, playerType, botCode, displayName}}
ludo.room.startingServer → AllRoom is full, game is starting. Payload: {room_id, seats[], started_with_bots}
ludo.game.snapshotServer → ClientPeriodic room state snapshot during waiting.
ludo.game.resultServer → ClientMatch completed. Includes settlement data and tournament status if applicable.
ludo.match.completeClient → ServerClient reports final match result (non-tournament).
ludo.errorServer → ClientError payload. Payload: {message}

8.2 Tournament-Specific Events

EventDirectionDescription
ludo.tournament.claim_roomClient → ServerClaim a tournament match room. Payload: {userId, accessToken, tournamentUuid, tournamentEntryUuid}
ludo.tournament.room_claimedServer → ClientRoom successfully claimed. Returns full room snapshot (human seats only — bots fill over time).
ludo.tournament.room_claim_failedServer → ClientClaim failed (not registered, tournament over, etc.). Payload: {message}
ludo.tournament.match_completeClient → ServerClient reports tournament match result. Payload: {tournament_uuid, tournament_entry_uuid, winner, placements[]}

8.3 Tournament Room ID Convention

Tournament rooms use a synthetic ID: "tournament-match-{match_id}" (e.g. "tournament-match-7").
This is NOT a real GameRoom UUID in the database — it's a string constructed by Laravel's claim-room endpoint.
Node.js and the internal start endpoint both detect this prefix to handle tournament logic separately.

12. Private Tournament Flow

12.1 How Private Tournaments Work

1
User creates tournament with type = "private". Laravel auto-generates a unique 6-character invite code (e.g. YAFWGH). Optional password field.
2
Tournament is NOT listed in GET /api/v1/tournaments (public list excludes private tournaments).
3
Creator shares the code (and password if set) with friends via external channel (WhatsApp, etc.).
4
Friend opens Unity → Tournament tab → "Join Private" button → enters code + password → Unity calls GET /api/v1/tournaments/private/{code}?password={pw}.
5
Laravel checks: invite code exists? Password matches? Returns tournament data.
6
Unity parses tournament → calls POST /{id}/register → then claim-room → socket join → game starts.

12.2 Private Tournament Limitations

  • No in-game invite link sharing yet (copy-to-clipboard feature not built).
  • No "My Private Tournaments" history screen in Unity.
  • Invite code is permanent — no expiry or one-time use option yet.

13. Prize Distribution Engine

13.1 Formula

Total Entry Pool = entry_fee × registered_human_players
Platform Fee (20%) = Total Pool × 0.20
Prize Pool (80%) = Total Pool × 0.80
Position Prize = Prize Pool × (prize_pct / 100)

13.2 Example — 8 players × ₹100 entry

ItemAmount
Total Entry Pool₹800
Platform Fee (20%)₹160 → credited to platform wallet
Prize Pool (80%)₹640
1st Place (60%)₹384
2nd Place (30%)₹192
3rd Place (10%)₹64

13.3 Distribution Rules

  • Runs automatically in distributePrizes() when all tournament matches complete.
  • Winners ordered by: status=winner first → then eliminated ordered by eliminated_at DESC (last-eliminated = highest rank).
  • Bots are skipped — prize cascades to next real player.
  • Each winner gets wallet credited instantly via Wallet model update.
  • A TournamentWalletTransaction row is created for every payout and for the platform fee.
  • Prize row gets payout_status = paid, winner_user_id, and paid_at.
  • Registration row gets final_position, prize_won, and completed_at.

  • Knockout bracket generation (2P and 4P per match)
  • Round Robin bracket generation
  • Queue-mode matchmaking (no bracket needed)
  • claim-room API (returns only humans, bot fill is server-driven)
  • Node.js socket claim flow (ludo.tournament.claim_room)
  • Bot fill (8s timer, one at a time, state: waiting_bot_fill)
  • Match result recording + bracket advancement
  • Prize distribution on tournament completion
  • Private tournament (invite code + optional password)
  • User-created tournaments via Unity (public & private)
  • Tournament list panel in Unity (compact fee-table style)
  • Private join popup in Unity

  • Simulation command (php artisan tournament:simulate)

15.2 Partially Done / Needs More Work ⚠️

  • Wallet deduction on register: Registration API exists and is idempotent, but actual wallet balance deduction and lock is not wired in TournamentRegistrationController yet (just records entry_fee_paid = 0).
  • Refund job: RefundTournamentRegistrations job exists but wallet credit logic not implemented.
  • Admin approve user tournaments: is_approved field exists but no admin approval UI or endpoint.
  • Match result from Node.js: The socket event ludo.tournament.match_complete is handled in LudoV2MatchmakingBridge.cs but needs more testing with real player accounts (currently tested with bots only).
  • Final position assignment: After prize distribution, final_position is set on the winner, but intermediate players (2nd/3rd) may not get their final_position correctly in all scenarios.
